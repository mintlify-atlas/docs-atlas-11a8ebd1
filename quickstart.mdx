This guide walks you through the `LightSwitch` example from the README, explains every piece, and then shows a second example with state-specific data.

## Build the LightSwitch machine

<Steps>
  <Step title="Define the states">
    Use `#[state]` on an enum to declare the legal lifecycle phases. Each variant becomes a distinct marker type:

    ```rust
    use statum::{machine, state, transition};

    #[state]
    enum LightState {
        Off,
        On,
    }
    ```

    `LightState::Off` and `LightState::On` are now separate types in the type system — not values of the same enum at runtime.
  </Step>

  <Step title="Define the machine">
    Use `#[machine]` on a struct to define the durable context that persists across all states. The first generic parameter must match the `#[state]` enum name:

    ```rust
    #[machine]
    struct LightSwitch<LightState> {
        name: String,
    }
    ```

    `LightSwitch<Off>` and `LightSwitch<On>` are **different types**. Fields like `name` are available in every state.

    <Tip>
      If you need derives such as `Debug` or `Clone`, place them **below** `#[machine]`:

      ```rust
      #[machine]
      #[derive(Debug, Clone)]
      struct LightSwitch<LightState> {
          name: String,
      }
      ```

      Placing derives above `#[machine]` causes a `missing fields marker and state_data` error.
    </Tip>
  </Step>

  <Step title="Define the transitions">
    Use `#[transition]` on `impl` blocks to declare the legal edges between states. Each transition method consumes `self` and returns the machine in its next state:

    ```rust
    #[transition]
    impl LightSwitch<Off> {
        fn switch_on(self) -> LightSwitch<On> {
            self.transition()
        }
    }

    #[transition]
    impl LightSwitch<On> {
        fn switch_off(self) -> LightSwitch<Off> {
            self.transition()
        }
    }
    ```

    `switch_on()` only exists on `LightSwitch<Off>`. `switch_off()` only exists on `LightSwitch<On>`.
  </Step>

  <Step title="Construct and use the machine">
    Build a new machine using the generated builder, then drive it through transitions:

    ```rust
    fn main() {
        let light = LightSwitch::<Off>::builder()
            .name("desk lamp".to_owned())
            .build();

        let light = light.switch_on();
        let _light = light.switch_off();
    }
    ```

    The builder requires you to supply every field declared on the machine struct. After `.build()`, the machine is in the state you specified as the generic parameter — here, `Off`.
  </Step>
</Steps>

## What the compiler prevents

Methods only exist on states where you define them. Calling a transition that doesn't exist for the current state is a **compile error**, not a runtime panic:

```rust
// This does not compile.
let light = LightSwitch::<Off>::builder()
    .name("desk lamp".to_owned())
    .build();

let _ = light.switch_off(); // error: no method `switch_off` on `LightSwitch<Off>`
```

The error appears at compile time, before your program ever runs. There is no guard code to write and no status field to check.

## State-specific data with the Checkout example

Some states carry data that is only meaningful in that phase. Use single-field tuple variants in your `#[state]` enum and `transition_with()` when moving into such a state.

The following example is from `lib.rs`:

```rust
use statum::{machine, state, transition};

#[state]
enum CheckoutState {
    EmptyCart,
    ReadyToPay(OrderDraft),
    Paid,
}

#[derive(Clone)]
struct OrderDraft {
    total_cents: u64,
}

#[machine]
struct Checkout<CheckoutState> {
    id: String,
}

#[transition]
impl Checkout<EmptyCart> {
    fn review(self, total_cents: u64) -> Checkout<ReadyToPay> {
        self.transition_with(OrderDraft { total_cents })
    }
}

#[transition]
impl Checkout<ReadyToPay> {
    fn pay(self) -> Checkout<Paid> {
        self.transition()
    }
}

fn main() {
    let cart = Checkout::<EmptyCart>::builder()
        .id("order-1".to_owned())
        .build();

    let ready = cart.review(4200);
    assert_eq!(ready.state_data.total_cents, 4200);

    let _paid = ready.pay();
}
```

Key points:

- `ReadyToPay(OrderDraft)` declares that this state carries an `OrderDraft` value.
- `transition_with(OrderDraft { total_cents })` passes the data into the next state.
- `ready.state_data.total_cents` accesses the state-specific data. The `state_data` field only exists on `Checkout<ReadyToPay>` — it is not present on `Checkout<EmptyCart>` or `Checkout<Paid>`.
- `pay()` uses `self.transition()` because `Paid` is a unit state with no associated data.

<Note>
  `state_data` is typed to the data you associated with that state variant. The compiler knows its shape and will reject mismatched access at compile time.
</Note>

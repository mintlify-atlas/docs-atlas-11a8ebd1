Showcases are not toy demos. Each one is a working, app-shaped program that wires Statum into a realistic stack — HTTP, CLI, SQLite, async workers, or WebSocket protocols — and demonstrates a specific pattern that matters in production.

Every showcase compiles and runs against the same workspace as the library. The source lives in `statum-examples/src/showcases/`.

## Running the showcases

Each showcase is a standalone binary in the `statum-examples` crate. Run any of them with:

```bash
cargo run -p statum-examples --bin axum-sqlite-review
cargo run -p statum-examples --bin clap-sqlite-deploy-pipeline
cargo run -p statum-examples --bin sqlite-event-log-rebuild
cargo run -p statum-examples --bin tokio-sqlite-job-runner
cargo run -p statum-examples --bin tokio-websocket-session
```

<Note>
  The `axum-sqlite-review` binary starts an HTTP server on `127.0.0.1:3000`. All other showcases run to completion and print their output, then exit.
</Note>

## What each showcase covers

| Showcase | Persistence | Key pattern |
|---|---|---|
| Axum + SQLite review | SQLite (in-memory) | Per-request typed rehydration via `#[validators]` |
| CLI deploy pipeline | SQLite (file-backed) | Multi-phase pipeline with branching outcomes |
| SQLite event log rebuild | SQLite (in-memory) | Append-only events, projection, batch `into_machines()` |
| Tokio job runner | SQLite (in-memory) | Lease-based locking, retry scheduling, async transitions |
| WebSocket session | None | Protocol-safe frame dispatch, no persistence needed |
